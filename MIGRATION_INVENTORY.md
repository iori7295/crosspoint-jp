## Category: FONT (font system, fallback, SD card fonts)

### lib/EpdFont/EpdFontFamily.cpp (we: +51, upstream: +5)

**Our additions:**
- `getGlyph()` — now walks a full fallback chain: primary → `fallbackFamily` → `globalFallback_` (with `glyphMissHandler` support)
- `getGlyphResident()` — like `getGlyph()` but never triggers `glyphMissHandler` (for measurement)
- `getDataForGlyph()` — returns font data that actually contains the glyph (walking fallback chain)
- Added `setFallback()`, `getFallback()`, `setGlobalFallback()`, `getGlobalFallback()` — static global fallback pointer
- Member: `fallbackFamily`, `globalFallback_` (static, inline)

**Upstream 1.5.0 changes:**
- `hasCodepoint()` — new method returning whether font covers a codepoint (via interval table)

**Overlap:** None — our changes add fallback chain infrastructure; upstream adds a single query method. Independent areas.

**Recommendation:** **keep** — both additions are needed.

**Porting notes:**
- Our `getGlyph()` fallback chain uses `globalFallback_` and `glyphMissHandler`, which upstream lacks. No conflict.
- Upstream's `hasCodepoint()` is a simple query; integrate alongside our changes.

### lib/EpdFont/EpdFont.cpp (we: +15, upstream: +23)

**Our additions:**

- Fixed a bounds bug in `getGlyph()`: added `cp >= interval.first` guard (was only checking `cp <= interval.last`)

**Upstream 1.5.0 changes:**

- Our `hasGlyph()`: interval-table only, no callback
- Upstream `hasCodepoint()`: interval table + `coverageHandler` callback
We should rename `hasGlyph` to match upstream's `hasCodepoint` and merge the logic.

**Recommendation:** **redesign** — merge our `hasGlyph()` with upstream's `hasCodepoint()`. Also verify our getGlyph() bounds fix is still needed.

**Porting notes:**
- Upstream `hasCodepoint` has different signature: includes `coverageHandler` dispatch. Our `hasGlyph` is used for the interval-only fast path. May need both.
- Upstream combining mark changes (`anchorFor`, `anchorOver`) replace the old combining mark math — ensure our vertical text layout engine is compatible.

**Upstream 1.5.0 changes (massive — ~80% of the file rewritten):**

- `SECTION_FILE_INCOMPLETE_VERSION` (=0) and `SECTION_FILE_PARTIAL_VERSION` sentinels

- `writeSectionFileHeader()` now takes `ReaderRenderSpec` instead of 10 individual params
- `loadSectionFile()` now takes `ReaderRenderSpec`, handles partial files, reads watermark trailer
- `finalizeBuild()` — new, promotes HTML cache and finalizes .bin
- `suspendBuild()` — persists partial build with parse watermark
- `abandonBuild()` — tears down without persisting
- `startBuild()` — new incremental build API
- `buildSomeMore()` — incremental page chunk builder
- `isBuildComplete()`, `isBuilding()`, `isPartial()`, `estimatedTotalPages()`
- `loadPageDuringBuild()` — read from in-progress build LUT
- `loadPageAt()` — read from disk file (finalized/partial)
- `loadPage()` — unified page resolver (tries build LUT first, falls back to disk)
- `loadPageFromSectionFile()` removed
- `getTextFromSectionFile()` — adapted to new block API (`wordCount()`, `wordText()`)
- `getCachedPageCount()` — rejects partial file version
- `findAnchor()` — new, resolves anchors from on-disk map
- `hasHtmlCache()` — HTML cache check
- `commitBuildFile()` — atomic swap of tmp .bin → final path
- `binTmpPath()` — tmp path helper
- `Page::serialize`/`deserialize` changes for v30-v32 format
- `buildOnly()` — `const` now
- Many internal state fields

**Overlap:** **Low** — our change is a small logging improvement in `loadSectionFile()`. Upstream rewrites the entire section build pipeline with incremental/windowed building, partial files, and new format. Our logging change is in a different part of `loadSectionFile()` (the parameter-mismatch check), which upstream also rewrites but the concept is the same.

**Recommendation:** **keep** — preserve the detailed log output when porting `loadSectionFile()`. The rest of upstream's changes are a massive architecture upgrade.

**Porting notes:**
- This will be the most complex file to port. The entire build pipeline is different.
- Our logging improvement should be added to upstream's version of the parameter-mismatch check.
- Our vertical layout engine (`VerticalSection`) is a separate class — not affected by upstream's Section changes directly, but may need `ReaderRenderSpec` adaptation.

- **SECTION API:** `loadSectionFile(10 params)` → `loadSectionFile(ReaderRenderSpec)`. Both our Section and VerticalSection need updating.
- **COMBINING MARKS:** Upstream changed the combining mark rendering API (`anchorFor`, `anchorOver`, `raiseAboveBase` with Anchor param). Our vertical text block may need corresponding changes.
